# 校园生存指北 - 项目说明文档

---

## 一、项目概述

### 1.1 产品定位

**校园生存指北 (Campus Survival Guide)** 是一款精细化校区地理信息系统（GIS），面向 B2B2C 场景，致力于实现校区内导航「最后一米」的精准交付。项目已从早期地图导航演进为**高精度校园运营仪表盘**，涵盖 POI 管理、集市交易、失物招领、活动、审核等完整闭环。

### 1.2 核心价值

| 价值点 | 说明 |
|--------|------|
| **精细化导航** | 基于高德步行路径 + 众包实时状态，导向至建筑具体入口、教室等校内细粒度点位 |
| **众包生态** | POI 留言板、实时状态上报（拥挤/施工/关闭），形成校园内信息聚合与共享 |
| **多租户隔离** | 基于 `schoolId` 的严格数据隔离，支持多所学校独立运营，可扩展性强 |

### 1.3 目标与指标

- **业务目标**：打造国内领先的精细化校区地理信息系统
- **核心指标**：导航任务完成率、POI 状态更新频次、多租户覆盖数、用户纠错采纳率

### 3.2 核心架构设计

#### 多租户隔离策略

- **数据层**：所有业务表含 `schoolId`，查询必须 `where: { schoolId }`（超管跨校除外）
- **权限层**：已绑定用户（Student/Admin/Staff）强制 `currentUser.schoolId`，不可切换；Guest/SuperAdmin 可切换
- **API 层**：非超管跨校请求返回 403

#### 图片存储与压缩

- Vercel Blob；image-utils 压缩 <1MB；image-upload 组件用于 POI 主图、头像

#### 邀请码停用逻辑

- DEACTIVATED 阻止关联用户登录；getCurrentUser/loginUser 校验；管理页仅停用/启用

#### 自适应 POI 抽屉（v1.3.1）

- 桌面端：右侧 `motion.div` 抽屉；移动端：vaul 手势 Bottom Sheet 0.35/0.85 吸附点；`useMediaQuery` 切换

#### 动态 POI 层级与 FitView

- 父子选择：`selectParentPOI`/`selectSubPOI`；`rootVisiblePois` 动态显隐；子 POI 简化视图
- FitView：选中时 `setFitView` 包围盒；`clearSelection()` 从 `mapViewHistory` 恢复

#### 扁平化 Marker 系统

- 24×24 圆形 `.flat-marker`；`--primary-theme` 变量；`offset: (-12, -12)` 锚点；40×40 聚合气泡

#### Split-View 布局

- admin-layout `overflow-y-auto`；campuses/pois 页 `overflow-hidden`，左右分区各自滚动

#### 认证与路由保护

- **HTTP Only Cookie**：`campus-survival-auth-token`，存储 userId、role、schoolId
- **Middleware**：仅拦截 `/admin/*`，校验 ADMIN/STAFF/SUPER_ADMIN
- **Super-Admin**：`/super-admin` 由页面内 `AuthGuard` 校验
- **角色矩阵**：SuperAdmin 负责基础设施（学校、全局分类、用户），不参与内容审核；Admin/Staff 负责运营（审核、活动、POI、校区）；审核侧边栏与 `/admin/audit` 对 SuperAdmin 隐藏/重定向

#### 数据访问与 100% Server Actions 架构

- **Server Actions**：业务 CRUD 100% 在 `lib/*-actions.ts`，统一返回 `{ success, data?, error? }`；多租户隔离在 Action 层强制 schoolId
- **API 路由**：仅保留认证（me/seed）、学校检测/列表/校区、公开分类、POI 搜索、审核、管理端、cron 等非 CRUD 场景
- **构建安全**：`useSearchParams()` 页面用 `<Suspense>` 包裹；依赖 `cookies()` 的 API 导出 `force-dynamic`

#### 构建与性能

- **Suspense 边界**：使用 `useSearchParams()` 的页面用 `<Suspense fallback={...}>` 包裹，保证静态分析时无 `dynamic` 泄漏，构建 100% 通过
- **force-dynamic**：依赖 `cookies()` 的 API 路由导出 `export const dynamic = 'force-dynamic'`，避免被静态预渲染

## 五、数据库模型摘要

### 5.1 核心表及 schoolId 分布

| 表名 | schoolId | 说明 |
|------|----------|------|
| School | - | 租户主体 |
| User / POI / CampusArea / Comment / LiveStatus / InvitationCode / LostFoundEvent / MarketItem / Activity | 必填 | 业务主表 |
| Category / CategoryOverride | 可选/必填 | 分类、覆盖 |
| MarketTransactionType / MarketCategory / MarketIntention / MarketLog | - | 集市类型、分类、意向、审计日志 |
| Notification | - | LIKE/REPLY/MENTION/SYSTEM/LOST_FOUND_FOUND |

- **Activity**：POI 关联的校内活动，schoolId、poiId、startAt/endAt
- **MarketLog**：集市交易审计，actionType 含 INTENTION_CREATED、ITEM_LOCKED、AUTO_UNLOCKED、AUTO_COMPLETED、TRANSACTION_COMPLETED 等

### 5.2 关键关联

- **POI** ↔ Category（categoryId）、POI 自关联（parentId → children）
- **Comment** ↔ User、POI、School；Comment 自关联（parentId → replies）
- **CommentLike** ↔ User、Comment
- **LiveStatus** ↔ POI、User、School
- **CampusArea** ↔ School
- **LostFoundEvent** ↔ School、POI、User
- **Notification** ↔ User（recipient、actor），entityId/entityType 关联 COMMENT/LOST_FOUND/POI
- **MarketLog** ↔ MarketItem（itemId）、User（userId），actionType 枚举记录意向创建、锁定、解锁、完成、管理员操作等

### 7.1 环境变量

```bash
# 数据库
DATABASE_URL="mysql://user:password@localhost:3306/campus_survival"

# 高德地图
NEXT_PUBLIC_AMAP_KEY="your_key"
NEXT_PUBLIC_AMAP_SECURITY_KEY="your_security_key"

# Vercel Blob 存储（POI 图片、用户头像）
BLOB_READ_WRITE_TOKEN="vercel_blob_rw_xxx"

NEXT_PUBLIC_APP_URL="http://localhost:3000"
```
