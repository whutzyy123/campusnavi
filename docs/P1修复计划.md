# P1 优先级问题修复计划

本文档对应《代码审查报告》中 **## P1（高）结构与架构问题** 的五项条目，目标是可迭代交付、低风险回归、可追溯验收。

**原则**：每项改动保持「单一目标 PR」或小步合并；每步前后运行 `npm run lint` 与 `npx tsc --noEmit`；涉及鉴权或多租户的改动需在 PR 描述中写明验证路径。

---

## 总览与推荐执行顺序

| 顺序 | P1 项 | 依赖 | 说明 |
|:---:|:---|:---|:---|
| 1 | 中间件鉴权链路优化 | 无 | 可先独立交付，减少对 `/api/auth/me` 的二次请求 |
| 2 | Auth 入口统一（getMe / 登出） | 与 1 可并行或紧跟 | 收窄「两套路径」中为关键的一条 |
| 3 | Market 兼容层收口 | 无硬性依赖 | 批量改 import，可脚本化校验 |
| 4 | API 响应契约收敛 | 可部分依赖 2 | 按域分批，优先高频与对外契约 |
| 5 | 超大文件拆分 | 与 3/4 可穿插 | 按业务优先级拆，避免与大重构同 PR |

## P1-1：中间件鉴权链路性能与稳定性

### 现状

- `middleware.ts` 通过 `fetchAuthMeRole` 内部 `fetch` `/api/auth/me` 解析角色。
- 风险：额外 RTT、`INTERNAL_APP_BASE_URL` / `NEXT_PUBLIC_APP_URL` / `VERCEL_URL` 配置错误时鉴权异常。

### 目标

- 在中间件链路内 **不发起 HTTP 自调用** 即可完成「有 Cookie → 是否为管理员路由所需角色」的判定。
- 保留与安全策略一致：**不信任客户端上传角色**，仅基于服务端会话与数据库（或等价已验证数据源）。

### 方案方向（选一为主，禁止长期双实现）

**推荐 A：`getAuthCookie` 侧增强供 Edge 不可用场景**

- Middleware 运行在 Edge：`lib/auth/server-actions.ts` 中若使用 Node-only API，需将「仅读 Cookie + 解析 session + 映射 role」抽成 **`lib/auth/session-edge.ts`** 或仅用 `NextRequest.cookies` + 共享的 JWT/不透明 token 校验（当前为 DB session token，则通常需 **Middleware 可调用的轻量校验**）。
- **若 Prisma 在 Middleware 不可用**：实现 `lib/auth/middleware-session.ts`，内联与 `getAuthCookie` **相同业务规则**（session 有效性、revokedAt、expiresAt、INACTIVE、`dbRoleToAppRole`、ADMIN/STAFF 邀请码停用检查），但通过 **可被 Edge 调用的数据访问方式**（如 HTTP 仅限内部、`unstable_cache` 不推荐用于鉴权）——**实际需要评估 Next 部署目标**：若为 Node middleware，可直接 import 共享校验函数调用 Prisma。

**执行前必选一步**：在项目当前 `middleware` 运行环境（Edge vs Node）下确认可否使用 `@/lib/core/prisma`；据此锁定 A 的子方案。

### 实施步骤

1. 阅读 `middleware.ts`、`lib/auth/middleware-admin-auth.ts`、`getAuthCookie` 与 `GET /api/auth/me` 的字段与分支，列出 **必须为 middleware 等价实现的不变量**（未登录、`INACTIVE`、`role` 非法、`DEACTIVATED` 邀请码等）。
2. 抽取 **纯函数** `resolveRoleForSessionToken(token: string): Promise<AppRole | null>`（或等价）到 `lib/auth/`，供 `getAuthCookie` 与 middleware **共用**，避免漂移。
3. 将 `fetchAuthMeRole` 改为调用上述共享函数（或移除 fetch 分支）。
4. 在 `staging` / 本地验证：`/admin/*`、`/super-admin/*` 登录、过期 Cookie、停用用户、停用邀请码、错误 `INTERNAL_*` 环境变量下的行为。
5. 可选：文档中写明生产环境必填 `INTERNAL_APP_BASE_URL` 的条款可降级为「仅开发兜底」。

### 验收标准

- 受保护路径进入时 **不再对 `/api/auth/me` 发起内部 HTTP 请求**（可通过日志或 temporary 计数断言）。
- 与改造前对比：四类场景（登录管理员、超管、`STAFF`、无 Cookie）行为一致。
- `npm run lint`、`npx tsc --noEmit` 全部通过。

### 风险与回滚

- 风险：Middleware 误用不完整校验导致越权或误拦。
- 缓解：对齐 `getAuthCookie` 的单测或可重复的手动 checklist；回滚保留 `fetch` 兜底开关（仅在短周期灰度中使用）。

## P1-2：核心入口统一（`getMe` / `/api/auth/me`；`logoutUser` / `/api/auth/logout`）

### 现状

- 用户信息：`lib/auth/server-actions.ts` 的 `getMe` 与 `app/api/auth/me/route.ts` 逻辑并行。
- 登出：`logoutUser` Server Action 与 `POST /api/auth/logout` 并行。
- Navbar 注释仍提及 `/api/auth/me`，但实际 `initializeAuth` 可走 `getMe`（易文档与代码漂移）。

### 目标

- **单一事实来源（SSOT）**：  
  - 读会话用户完整信息：**仅一处实现**（建议保留 `getMe` 内核心查询 + 由 Route 薄封装或直接废弃 Route）。  
  - 清会话：**仅一处实现** `removeAuthCookie` + redirect 策略；HTTP Route 仅存「无法用 Action 的场景」兜底并文档化。
- 对外 **`/api/auth/me`**：若 Middleware 已无依赖且客户端全迁 `getMe`，可 deprecate Route 或在 Route 内 **一行调用共享函数**，禁止双份查询逻辑。

### 实施步骤

1. 抽出 `buildMeResponse(auth: AuthCookieData): Promise<GetMeResult>` 或 `loadMeUserByAuth(auth)` 到 `lib/auth/me-shared.ts`（文件名按项目惯例）。
2. `getMe()` 与 `GET /api/auth/me` **仅调用该模块**，删除重复 `findUnique` 与分支。
3. 登出：`logoutUser` 保持主路径；`/api/auth/logout` **仅调用** `removeAuthCookie()`，与现状一致但要与 Action 共用同一辅助函数并在文件头注释「兜底场景列表」。
4. 全局 `grep`：`/api/auth/me`、`fetch\(.*auth/me`，统一替换为 `getMe`（Client 组件内）或服务端调用共享模块。
5. 更新 `.cursor/rules` 或 `docs/API.md`（若有）中 auth 小节，写明唯一推荐路径。

### 验收标准

- `GET /api/auth/me`（若保留）与 `getMe()` 返回值字段与错误码语义一致。
- 无重复的 `user.findUnique({ id: authData.userId })` 长段逻辑散落在两处。
- 客户端初始化仅保留一种主推方式（另一种是兼容层则需标 `@deprecated` 与移除时间表）。

## P1-3：兼容层收口（以 `lib/actions/market.ts` 为先行）

### 现状

- `lib/actions/market.ts` 为 `@deprecated` 转发层，仍有大量 `@/lib/actions/market` 引用。
- `lockMarketItem` 等仍通过兼容导出。

### 目标

- 新代码与存量代码统一从 **`@/lib/market`** 或 **`@/lib/market/index`** 按领域子模块导入。
- `lib/actions/market.ts` 在过渡期保留 **重新导出别名**（可选），但 **eslint 禁止新引用**（`no-restricted-imports`）或 Codemod 一次性替换。

### 实施步骤

1. `grep -r "@/lib/actions/market"` 列出全部文件；按目录分批（优先 `app/api`、`components/market`、`lib/market/*`）。
2. 对每个文件：`import X from '@/lib/market'` 或对等子路径 `item-admin`、`transaction` 等（以现有 `lib/market/index.ts` 导出为准）。
3. 全量替换完成后：
   - 将 `lib/actions/market.ts` 改为单行 re-export **并** 加 `// @deprecated — 计划于 YYYY-MM 删除`；
   - 或删除文件并修正所有 import（推荐在批次末尾执行）。
4. 运行 `npm run lint` / `tsc`；冒烟：集市发布、交易流、cron `market-deadlock`、管理员下架接口。

### 验收标准

- 仓库内无业务代码 `from "@/lib/actions/market"`（或仅存允许列表）。
- Market 相关功能冒烟通过；无循环依赖报错。

## P1-4：API 响应契约收敛

### 现状

- 并行形态：`success + data`、`success + users`、`success + schools`、顶层兼容别名（如 `/api/schools`）。
- 增加前端与消费者心智负担。

### 目标（分阶段）

**阶段 A（不立旧破）**：为新响应增加 **规范形态**，旧字段保留一至两个版本：

- 成功：`{ success: true, data: T, message?: string }`
- 分页：`data` 内含或并列 `pagination`（与现有 `getPaginationMeta` 对齐）。
- 错误：`{ success: false, message: string, error?: string }`（与 `jsonErr` 一致）

**阶段 B**：前端只读 `data`；移除顶层 `users`、`schools` 等别名（Breaking change，需版本或协调）。

### 实施步骤

1. 建立「契约表」：`docs/` 或小表记录在 `lib/api/contracts.md`（**若团队禁止新 md 则可放 PR 正文**）：列出 `/api/users`、`/api/schools`、`/api/admin/users`、keywords 等目标形态与时间线。
2. 按路由批次修改：`jsonOk({ data: { users } })` 替代 `NextResponse.json({ success, users })`，并临时保留 `users` 顶层别名 + `console.warn` 或 telemetry（可选）。
3. 前端 `fetch`：统一解构 `payload.data`；删除对旧字段分支。
4. 删除兼容别名 PR（独立、可回滚）。

### 验收标准

- `lib/api/http.ts` 的 `jsonOk`/`jsonErr` 成为新常态；主要 admin/super-admin 列表接口结构一致。
- 移动端/旧客户端如有，需兼容性说明或版本号。

### 优先级建议

- 先统一 **admin 控制台高频接口**（用户、学校、关键词），再长尾。

## P1-5：超大业务文件拆分

### 清单（来自审查报告，可随 `wc -l` 更新）

| 文件 | 建议拆分粒度 |
|:---|:---|
| `components/poi-drawer.tsx` | `usePoiDrawerState`、`PoiDrawerHeader`、`PoiDrawerTabs`、地图/markdown 段落 |
| `app/admin/school/campuses/page.tsx` | 数据加载与权限、`CampusTable`、`CampusEditorDialog` |
| `lib/actions/comment.ts` | 按「列表/审核/写入/校验」拆到 `lib/actions/comment/` 或 `lib/comment/` + actions 薄层 |
| `components/poi-map.tsx` | AMap 生命周期 hook、`MarkersLayer`、`usePoiMapSelection` |
| `lib/actions/category.ts` | 读/写/管理员/合并规则分文件 |

### 实施步骤（每个文件遵守同一范式）

1. **先提取纯函数与类型**（无 `"use server"`）到 `lib/...`，保证可测。
2. **再提取 hooks**（`hooks/use-*.ts`）减少 `useEffect` 堆叠。
3. **最后抽展示组件**，`page.tsx` 只拼装与路由相关逻辑。
4. 单文件 PR：一次只拆一个巨无霸，避免与市场/契约大改混提交。

### 验收标准

- 单文件行数：**目标 ≤ 400 行**（与项目规则对齐），或拆分后单一职责清晰且在 PR 中说明例外。
- 行为与网络请求次数不明显劣化（尤其地图场景）。
