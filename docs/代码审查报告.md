# CampusProject 代码审查报告

生成时间：2026-05-13  
审查范围：`app`、`components`、`lib`、`middleware.ts`、`prisma/schema.prisma`、`docs`  
审查方式：目录结构扫描 + 关键路径抽样阅读 + 静态检查（`next lint`、`tsc --noEmit`）

---

## 一、执行摘要（高优先级结论）

当前项目处于一次大规模重构/迁移进行中，代码可通过 lint 与类型检查，但存在明显的结构与架构债务，主要风险集中在：

1. **模块迁移未完全收口**：存在大量“兼容层 + 新结构并存”的中间态，后续易继续扩散技术债。  
2. **核心模块体积过大、职责过多**：多个超大文件承担过多领域逻辑，调试与回归成本高。  
3. **API 契约风格不统一**：同类接口返回结构差异较大，前端调用复杂度和误用概率上升。  
4. **Debug 可观测性不足**：大量空 `catch` 或吞错处理，问题定位依赖人工复现。  
5. **文档资产存在丢失风险**：`docs` 目录出现多份空文档/替换痕迹，知识沉淀不稳定。

---

## 二、已确认问题（按严重度）

## P1（高）结构与架构问题

- **重构处于“新旧并存”状态，兼容层未收口**  
  - `lib/actions/market.ts` 明确标注为转发层且带 `@deprecated`，但项目内仍被大量模块直接依赖（API、页面、组件均有调用）。  
  - 风险：迁移周期被拉长，调用路径不透明，后续新代码继续依赖旧入口。

- **业务文件过大，单文件承担多个关注点**  
  - 已观测到多个超大文件：`components/poi-drawer.tsx`、`app/admin/school/campuses/page.tsx`、`lib/actions/comment.ts`、`components/poi-map.tsx`、`lib/actions/category.ts` 等。  
  - 风险：修改局部需求时难以保证全局不回归，代码评审和单点定位成本高。

- **核心入口并存两套调用路径（Server Action 与 API Route）**  
  - 例如用户信息链路同时存在 `getMe` 与 `/api/auth/me`；登出链路同时有 `logoutUser` 与 `/api/auth/logout`。  
  - 风险：逻辑漂移（字段、错误语义、鉴权行为不同步），增加维护与排障复杂度。

- **API 响应契约不统一**  
  - 存在 `{ success, data }`、`{ success, users }`、`{ success, schools }`、`{ success, data + 顶层兼容别名 }` 等并行形式。  
  - 风险：前端消费层需要写额外分支，接口复用困难，易出现“同域不同格式”的隐性 bug。

- **中间件鉴权链路性能与稳定性敏感**  
  - `middleware.ts` 通过 `fetchAuthMeRole` 间接调用 `/api/auth/me` 获取角色。  
  - 风险：每次受保护路由访问引入额外网络/序列化开销；内部 URL 配置错误时会导致鉴权链路抖动。

## P2（中）目录与工程组织问题

- **`docs` 目录文档完整性风险明显**  
  - 当前可见多份 `docs/*.md` 为 0 字节（如 `API.md`、部分中文文档）。  
  - 风险：架构知识、规范、数据口径无法作为可靠“单一真相源”。

- **目录重构粒度不一致**  
  - `lib` 已迁移到分域目录（`auth`/`core`/`content`/`geo`/`market`），但仍存在大量历史入口与过渡性导出。  
  - 风险：新同学难以判断“标准入口”，容易重复造轮子或引入错误 import。

- **UI 组件层规模增长快，边界定义不够清晰**  
  - `components` 下既有通用组件，也有高度业务耦合复合组件，且复用层次不统一。  
  - 风险：UI 变更时牵一发动全身，难做稳定抽象。

## P2（中）代码逻辑与可维护性问题

- **空 `catch` / 吞错模式较多，影响 Debug 闭环**  
  - 在地图、POI、市场、若干页面中存在 `catch {}` 或仅降级处理不记录上下文。  
  - 风险：线上故障难以定位根因，尤其是异步链路和地图 SDK 边界问题。

- **路由参数模式不一致（动态路由 params 类型写法混用）**  
  - 大多数路由使用常规 params 对象，个别路由使用 `Promise<{ id: string }>` 形式。  
  - 风险：可读性与维护一致性下降，后续迁移/脚手架生成易出偏差。

- **角色与状态存在“未知值兜底字符串”传播**  
  - 部分用户列表逻辑对未映射角色返回 `"UNKNOWN"`。  
  - 风险：上游 schema 异常会在前端层扩散，若 UI 未显式处理会产生不可预期展示/权限判断分支。

- **错误消息语义与语言风格不完全一致**  
  - 同域接口/Action 中中文、英文、业务码表达混用。  
  - 风险：前端统一提示与监控聚合（按 message 分类）效果变差。

## 三、潜在风险（当前未必已触发故障）

- **缺少自动化测试护栏**  
  - 未发现测试文件（单元/集成/e2e）。  
  - 风险：在当前大规模重构背景下，回归风险高度依赖人工验证。

- **限流表增长的运维风险**  
  - `RateLimit` 采用数据库固定窗口计数，未见明确清理策略。  
  - 风险：长周期运行后表体膨胀、索引效率下降，影响高频接口性能。

- **`force-dynamic` 使用范围较大**  
  - 多个 API 路由显式强制动态。  
  - 风险：缓存收益降低，吞吐与成本对请求波动更敏感。

- **重构阶段分支差异非常大**  
  - 当前工作树包含大量 `M/D/??`，涉及 `app`、`components`、`lib`、`docs` 全域。  
  - 风险：一次性合并冲突与隐性回归概率高，评审难度上升。

---

## 四、建议的治理优先级（建议按迭代执行）

1. **先收口入口**：为每个域明确唯一“对外入口”（Action 或 API），标注并逐步移除兼容层。  
2. **再拆大文件**：优先拆分 >700 行文件，按“状态管理 / 数据请求 / 视图渲染”三层切分。  
3. **统一契约**：全局统一 API 返回格式（建议 `success + data + message + pagination`），并清理兼容别名。  
4. **补 Debug 基建**：替换关键空 `catch`，补充结构化错误日志（场景、参数、用户、schoolId、requestId）。  
5. **建立测试底座**：先覆盖鉴权、租户隔离、关键交易状态机（market/comment/poi）的最小回归集。  
6. **修复文档资产**：恢复 `docs` 空文件内容，建立“文档变更必须含有效内容”的校验。

---

## 五、本次检查结论

项目具备较好的类型与 lint 基线（当前可通过），但从工程化视角看，**主要风险不在语法层，而在重构中间态导致的架构一致性与可演进性**。  
建议优先处理“入口收口 + 超大文件拆分 + 契约统一 + Debug 可观测性”四项，否则后续功能迭代将持续放大维护成本与回归风险。
