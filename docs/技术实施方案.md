《校园生存指北》技术实施方案 (Technical Implementation Plan)
一、 系统逻辑架构
本系统采用分层架构，确保多租户隔离与地理信息的高效处理。
接入层 (Access Layer)：Next.js Middleware。负责校验用户所属 schoolId，拦截非校内定位。
业务逻辑层 (Service Layer)：
SchoolService: 处理围栏算法与校区初始化。
NaviService: 负责高德路径规划与自定义校内小径的融合。
CrowdService: 处理 POI 状态的实时更新与权重计算。
持久化层 (Data Layer)：Prisma ORM。负责与 MySQL 进行类型安全的交互。
二、 核心数据库设计 (Database Schema)
针对多租户与导航需求，我们将建立以下核心表结构：
1. 学校表 (School)
存储校区物理边界。
字段：id, name, boundary (JSON 格式的多边形顶点坐标), center_lat, center_lng (地图初始化中心)。
2. POI 资源表 (POI)
字段：id, schoolId, name, category (食堂/教学楼等), lat, lng, isOfficial (布尔值), entryPoints (JSON，存储建筑的具体入口坐标点)。
3. 校内路径网格表 (Route_Edges) - 核心导航支持
为了补全高德地图不识别的小路。
字段：id, schoolId, start_node (JSON坐标), end_node (JSON坐标), weight (权重：距离/拥堵度), type (步道/阶梯/坡道)。
4. 实时状态表 (Live_Status)
字段：id, poiId, statusType (拥挤/营业), val, userId, createdAt, expiresAt (过期时间)。
三、 核心模块技术实现路径
3.1 地理围栏碰撞算法 (Geofencing)
挑战：如何高效判定用户是否在校内？
实现：
前端：使用高德 SDK 的 AMap.GeometryUtil.isPointInRing 方法进行初步判定。
后端：使用 Turf.js 库在 API 层面二次校验，防止非法数据通过接口注入。
3.2 深度导航与路径规划 (Advanced Navigation)
逻辑方案：
调用高德规划：首先调用高德步行导航 API。
自定义路径叠加：如果终点是校内特定 POI（如“3号教学楼后门”），逻辑会自动将高德终点设为最近的校内已知路径点，随后通过内部算法接管“最后 100 米”的引导。
AR 方向指引：通过手机陀螺仪 API（Web Device Orientation），在地图上实现类似“指南针”的导航体验。
3.3 众包数据的“衰减与信誉”算法
实现：
实时性：POI 的“拥挤度”采用 加权平均法。最新的上报权重为 0.7，前 30 分钟上报权重为 0.3。
自动失效：数据库设置定时任务（Cron Job）或在查询时通过 WHERE expiresAt > NOW() 逻辑实时过滤过期信息。
四、 接口设计 (API Specification)
模块	接口路径	方法	说明
校区	/api/school/detect	POST	传入经纬度，返回匹配的 schoolId 和边界
地图	/api/pois/list	GET	根据 schoolId 和范围拉取所有点位
众包	/api/pois/report	POST	L2以上用户提交实时状态
导航	/api/nav/path	GET	结合高德数据与本地路径网格，返回最优路径
风控	/api/audit/report	POST	提交举报内容
五、 安全与合规实施
内容安全审核 (DFA 算法)：
在 lib 目录下封装 ContentReview 类。
使用本地字典进行词法扫描，确保 UGC 内容（POI 名称、动态）不包含敏感词。
数据脱敏：
所有 API 返回的用户信息仅包含 nickname 和 avatar，严禁返回 phone 或精确的实时 location 序列。
频率限制 (Rate Limiting)：
针对 /api/pois/report 接口，基于 IP 和 UID 设置限流（如：1分钟内仅允许提交 1 次状态更新）。
