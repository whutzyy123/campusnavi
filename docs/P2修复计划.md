# P2 优先级问题修复计划

本文档对应《代码审查报告》中 **P2（中）** 两类共 **七项**：「目录与工程组织」三项、「代码逻辑与可维护性」四项。目标是在不阻塞 P1（架构/契约/鉴权）的前提下，**提升可发现性、一致性与可观测性**，可分多 PR 迭代完成。

**原则**：单 PR 单一主题；每步 `npm run lint` 与 `npx tsc --noEmit`；与 P1「API 契约」「错误码」冲突时，以 P1 决策为准并在此文档中追加「对齐说明」。

---

## 总览与推荐执行顺序

| 顺序 | P2 项 | 依赖 | 说明 |
|:---:|:---|:---|:---|
| 1 | Docs 文档完整性 | 无 | 成本低，恢复团队「单一真相源」 |
| 2 | `lib` 标准入口与过渡期规则 | 可与 1 并行 | 减少错误 import，可与 eslint 规则捆绑 |
| 3 | UI 组件分层约定 | 独立 | 渐进式整理目录与命名，不要求一次大搬迁 |
| 4 | 路由 `params` 写法统一 | 无 | 以小批次改 `app/api/**/route.ts`，与 Next 版本一致 |
| 5 | `UNKNOWN` 角色与类型收窄 | 可选依赖 P1 角色策略 | 后端列表过滤或前端显式分支 |
| 6 | 错误文案 / 语言一致 | 可与 P1-4 对齐 | 优先用户可见字符串；内部日志可保留英文 |
| 7 | `catch` 吞错治理 | 贯穿式 | 按域名优先级（地图/POI/市场 → 其他）分批替换 |

说明：4、5、6、7 可与 P1 并行，但 **6 若在 P1 已引入统一 `message`/`error` 结构，应直接复用**，避免二次修改。

## P2-A：目录与工程组织

### P2-A1：`docs` 目录文档完整性

#### 现状

- 多份 `docs/*.md` 为 **0 字节** 或缺失实质内容，架构与规范不可依赖。

#### 目标

- `docs` 下所有**被引用或列入 README 索引**的文档均有有效内容；新增「空文件」不可合并（可选 CI 防护）。

#### 实施步骤

1. **盘点**：列出 `docs/` 全部 `.md`、字节数、是否与 `README`、`PRD`、规则文件互链。
2. **恢复**：对空文件从 git 历史恢复最后有效版本，或按当前代码重写最小可用章节（API 路径表、环境变量、角色说明）。
3. **索引**：在 `README.md` 或 `docs/README.md`（若采用）中维护文档目录，避免孤儿文件。
4. **可选护栏**：CI 或 `pre-commit` 脚本检查 `docs/**/*.md` 非空且字数大于阈值（例如 ≥ 200 字符，排除纯占位符）。

#### 验收标准

- 无 0 字节生产文档；团队成员能仅通过 `docs` 找到 API/规范/数据口径入口。
- （若启用检查）主干流水线对空文档失败。

### P2-A2：`lib` 目录重构粒度与「标准入口」

#### 现状

- 已存在 `auth`、`core`、`content`、`geo`、`market` 等子域，但与 **历史路径、兼容转发、多入口导出** 并存，新代码易选错 import。

#### 目标

- 文档 + 工具双轨：明确 **推荐使用路径**；过渡期用 lint 限制 **`@deprecated` 转发层** 的新增引用。

#### 实施步骤

1. **立表**（可放在 `docs/开发规范.md` 或本节附录）：域 → 首选 import 路径 → 禁止/弃用路径示例（如 `@/lib/actions/market` 待迁出后标弃用）。
2. **ESLint**：对已知弃用模块配置 `no-restricted-imports`（与 P1 Market 收口里程碑对齐，避免重复劳动）。
3. **Barrel 导出**：各子域 `index.ts` 仅导出稳定 API，避免深层路径泄露实现细节；必要时 `lib/README` 一段说明。
4. **清理**：在 P1 收口完成后删除已无引用的转发文件，并更新本计划状态。

#### 验收标准

- 新 PR 默认不引入新增弃用路径；Code Review 有可查「标准入口」依据。

---

### P2-A3：UI 组件层边界（`components/`）

#### 现状

- 通用 UI 与强业务复合组件混在同一顶层，复用层次不统一。

#### 目标

- 约定分层与命名，**渐进搬迁**；不要求一次性移动所有文件。

#### 推荐约定（可按项目习惯微调）

| 层级 | 路径示例 | 职责 |
|:---|:---|:---|
| 设计系统 / 原子组件 | `components/ui/*` | Button、Input、Modal 壳、无业务文案耦合 |
| 领域可复用块 | `components/{domain}/*` | 如 `market/`、`activity/`，可含业务文案与接口形状 |
| 页面专用 | 紧邻 `app/**/page.tsx` 的局部组件或 `components/page-*` | 仅单路由使用则考虑 colocation |

#### 实施步骤

1. 在 `.cursor/rules` 或 `docs/开发规范.md` 中写入上表与「何时必须拆到 `components/ui`」的门槛（例如同一控件第 3 次复用）。
2. 每次改大文件或新功能：**新代码**必须符合约定；旧文件用「触达即搬」策略小步迁移。
3. 与 P1「超大文件拆分」配合：拆出的纯展示块优先落在 `components/ui` 或领域子目录。

#### 验收标准

- 评审时有明确「应放哪一层」的依据；新增巨型 `components/*.tsx`（如单文件 >600 行）在 PR 中被挑战或附拆分计划。

## P2-B：代码逻辑与可维护性

### P2-B1：空 `catch` / 吞错模式

#### 现状

- 地图、POI、市场等处存在 `catch {}` 或仅静默降级，**无法从日志还原上下文**。

#### 目标

- 关键路径：**记录**（`console.error` 或统一 `logger`）+ **用户可感知降级**（toast / 占位 UI）；极少数预期错误可 narrowly catch 并注释原因。
- SDK 边界（如 `map.remove`）：允许空 catch，但须在 **单行注释** 说明「为何可忽略」。

#### 实施步骤

1. **策略文档**：一页「Catch 指南」——禁止裸 `catch {}` 的目录列表（如 `lib/actions`、`app/api`）；允许的例外与白名单写法。
2. **工具**：若项目暂无 logger，可先封装 `lib/core/log.ts` 或沿用 `console.error` 统一前缀如 `[poi-map]`。
3. **分批修复**：优先级 **Server Action / Route Handler → 市集/地图客户端 → 其他**。
4. **回归**：每条 PR 附带「如何人为触发错误仍能看见日志/提示」的一句话。

#### 验收标准

- `app/api`、`lib/actions` 中无无注释的空 `catch`；客户端高危路径仅剩有注释的白名单。
- 抽检：伪造网络失败仍可看到错误或日志关键字。

### P2-B2：动态路由 `params` 类型写法统一

#### 现状

- 多数路由为 `params: { id: string }`，个别为 `Promise<{ id: string }>`（与未来 Next 版本或误判混用）。

#### 目标

- 与 **仓库当前 Next 版本**官方 Route Handler 签名一致；全库同一风格。

#### 实施步骤

1. **读官方**：查阅当前 `next` 主版本（`package.json`）对 `app` 路由 `params` 的推荐类型。
2. **基线选一个**：若为同步对象，则把所有 `Promise<params>` **改为同步**并 `await`/解构一致；若为异步 Promise，则反向统一。
3. **机械替换**：逐个 `route.ts` 小 PR 修改；改后跑 `tsc`。
4. **贡献者提示**：在 `docs/开发规范.md` 中贴一条正确模板，避免再次出现分歧。

#### 验收标准

- `grep` `params:\s*\{\s*params:\s*Promise`（或等价）结果为 0 或仅剩有意文档化例外。
- TypeScript 无相关类型告警。

### P2-B3：`UNKNOWN` 角色兜底字符串传播

#### 现状

- 管理员用户列表等对 `dbRoleToAppRole` 失败时使用 `"UNKNOWN"` 字符串外露。

#### 目标

- **类型**：前后端对消费者暴露 `AppRole | "UNKNOWN"` 时须有显式分支，或后端 **过滤/拒绝返回**非法行。
- **产品**：管理端列表对异常数据展示「数据异常」并引导修复，而非当作普通角色参与筛选。

#### 实施步骤

1. **数据来源**：核对 `schema` 中 `User.role` 合法值与实际种子数据；确认为 0「游客」是否与列表预期一致。
2. **服务端**（任选或组合）：  
   - 列表查询 `where role in (1,2,3,4)` 排除非法；  
   - 或对 `null` 映射导出为显式 **`INVALID_ROLE`** + `roleNumber`，UI 单列展示警告。  
   - 避免静默当普通职员处理权限。
3. **类型收窄**：TS 上使用联合类型或 `zod` parse 出站 DTO。
4. **监控**：可选对 `UNKNOWN`/`INVALID` 出现次数打日志或埋点。

#### 验收标准

- 超管列表中不再出现无声的 `"UNKNOWN"` 混在角色筛选逻辑中且无 UI 提示；或直接不返回该行并记录审计日志（按产品选择并在 PR 中说明）。

### P2-B4：错误消息语义与语言风格一致

#### 现状

- 同域中英文混用、`message` 与 `error` 字段语义不统一。

#### 目标

- **用户可见**：统一中文（或统一走 i18n key，二期可做）。  
- **机器可读**：HTTP 状态码 + 可选 **`code`** 字段（与 P1 API 契约统一时一并落地更高效）。  
- **内部**：`error`/`stack` 不向前端泄漏生产环境细节。

#### 实施步骤

1. **与 P1 对齐**：若已采用 `jsonErr(message, status, error?)`，则约定 `message` 为用户文案，`error` 为开发/日志用简短码。
2. **审计批次**：登录注册 → 集市 → POI → 管理端——每批只做字符串与 code 规整，不改业务分支。
3. **禁止**：用户可见英文句与中文句随意混在同一路由；注册等已有英文句统一翻译或改用 code。
4. **文档**：列出常见 `code` 表（可与 `docs/API.md` 合并）。

#### 验收标准

- 同一功能域抽检：用户提示语种一致；404/401/403 文案可被产品一句话确认。
-（若引入 `code`）前端可按 `code` 分支，减少对中文 `includes` 的脆弱判断。

## 里程碑建议（可与 P1 穿插）

| 里程碑 | 内容 |
|:---|:---|
| P2-M1 | Docs 恢复 + lib/UI 分层写入规范 |
| P2-M2 | `params` 全库统一 + UNKNOWN 策略落地（后端或 UI） |
| P2-M3 | Catch 指南 + 高风险路径首轮清理 |
| P2-M4 | 错误文案与 code 首轮审计（与市场/管理员域优先） |

## PR 自检清单（P2 适用）

- [ ] 未引入新的弃用 import（与 A2 规则一致）。
- [ ] 新 catch 附带注释或可观测日志（与 B1 一致）。
- [ ] `params` 与当前 Next 推荐签名一致。
- [ ] 用户可见文案语言策略符合 B4。
- [ ] Lint + TypeScript 通过。
