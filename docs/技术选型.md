《校园生存指北》技术选型文档 (Tech Stack Specification)
一、 整体架构图 (System Architecture)
采用 BFF (Backend For Frontend) 模式，利用 Next.js 实现前后端同构，简化部署逻辑。
前端 (Client)：Next.js (React) + Tailwind CSS + Lucide Icons
后端 (Server)：Next.js API Routes (Node.js)
数据库 (Database)：MySQL 8.0 + Prisma ORM
地理信息服务 (GIS)：高德地图 JS API 2.0 + Turf.js (几何计算)
二、 详细选型方案
1. 前端框架：Next.js 14 (App Router)
理由：
SEO 与性能：校园产品未来若转 Web 端，SSR/ISR 渲染能极大提升首屏速度。
全栈一体化：API 路由直接写在 app/api 目录下，无需独立搭建 Express 以后端，适合 Cursor 快速联调。
生态：目前 AI 对 Next.js 14 的 App Router 模式掌握最为精准。
2. 样式与 UI 库：Tailwind CSS + Shadcn UI
理由：
Tailwind：通过类名直接写样式，AI 生成页面布局的速度比手写 CSS 快 5 倍。
Shadcn UI：这是目前大厂流行的组件库方案，代码完全开放，Cursor 可以根据需要直接修改组件源码。
3. 数据库与 ORM：MySQL 8.0 + Prisma
理由：
MySQL：你已经配置好了，其空间索引（Spatial Index）足以支撑校园级的 POI 查询。
Prisma ORM：重点推荐。它提供强大的类型安全。你只需定义 schema.prisma 文件，AI 就能自动推断出所有的数据库操作逻辑，极大地减少了 SQL 注入风险和手动写 SQL 的痛苦。
4. 状态管理：Zustand
理由：
比 Redux 轻量，比 React Context 性能好。非常适合存储“当前学校 ID”、“用户信息”、“导航状态”等全局数据。
5. 地理计算引擎：Turf.js
理由：
校内导航核心：高德 API 有时无法覆盖极细的校内小径。Turf.js 可以在前端进行复杂的几何计算（如：判断点是否在多边形内、计算两点间折线距离、路径平滑处理）。
三、 核心技术点攻克方案（PM 经验谈）
1. 多租户数据隔离方案 (Multi-Tenancy)
方案：逻辑隔离（Shared Schema）。
实现：所有表增加 schoolId 字段。
中台化设计：在服务端中间件（Middleware）统一拦截请求，从 Header 或 Session 中获取 schoolId，确保所有查询自动带上租户过滤。
2. 校内高精度导航方案
挑战：高德官方路径规划不识别校内“林间小道”。
选型：
数据结构：采用 GeoJSON 格式存储校内特有的“路径网格”。
算法：当高德导航无法覆盖时，调用后台存储的校内路径点（Node）和边（Edge），使用 Dijkstra 算法 或 A 算法* 进行前端路径规划补全。
3. 地理位置平滑 (Location Smoothing)
挑战：校园教学楼密集，GPS 信号容易跳跃。
选型：使用 卡尔曼滤波 (Kalman Filter) 算法处理高德地图返回的坐标，防止导航时小蓝点“乱跳”。
