一、 项目综述 
1.1 项目背景
现状描述：通用地图（高德/百度）对校区内部的覆盖粒度不足，缺乏校内小路、建筑出入口、实时排队状态等精细化数据。
痛点分析：新生入校找路难、外卖/快递点定位模糊、校内临时封路或施工信息更新滞后。
1.2 项目目标与核心价值
业务目标：打造国内领先的精细化校区地理信息系统（GIS），实现校区内导航“最后一米”的精准交付。
用户价值：提供基于众包的实时校内路况、精准建筑入口导航及校内资源分布图。
1.3 核心指标
北极星指标：导航任务完成率（用户发起导航后到达终点的比例）。
过程指标：POI 状态更新频次、多租户（学校）覆盖数、用户纠错采纳率。
二、 用户分析
2.1 目标用户画像
新生/访客：对校内环境陌生，强依赖地图导向。
认证学生/校内常驻人员：熟悉环境，但需要实时情报（如：食堂排队、自习室拥挤度、临时封路）。
学校管理员 (Campus Admin)：负责维护官方地理数据及校内公告。
2.2 用户场景描述
场景 A (精准寻址)：新生需要去往“公教楼 302”，地图需导向至该楼最靠近该教室的具体入口。
场景 B (生存避坑)：用户想去二食堂，查看地图发现其他同学标记“排队极长”，遂改去三食堂。
场景 C (路线规划)：用户需要在两节课之间快速转场，系统规划出一条避开“施工路段”或“人群密集区”的校内小捷径。
三、 业务逻辑与流程图 (Logic & Flow)
3.1 业务架构图 (Architecture)
数据层：基于 MySQL 的多租户存储（School_ID 隔离）。
服务层：高德 API 二次封装、校内路径权重计算引擎、众包状态衰减逻辑。
应用层：Web App (Next.js) + 管理后台。
3.2 核心导航流程图 (Business Flow)
起点/终点输入 -> 2. 获取校内路径点集 -> 3. 计算实时路况权重（如：若某段路被标记施工，则权重无限大） -> 4. 路径渲染 -> 5. 实时位置偏移校正 -> 6. 到达目的地检测。
3.3 状态机图 (POI Status)
POI 状态：官方认证 <-> 众包更新 -> 过期自动重置 -> 管理员修正。
四、 功能需求详述 (Functional Requirements)
4.1 模块 A：多租户地理围栏与初始化
4.1.1 功能定义：基于地理围栏的学校自动识别与数据隔离。
4.1.2 详细规则：
地理围栏判定：采用射线法（Ray-casting）判断用户 GPS 坐标是否在学校 Boundary 多边形内。
多租户隔离：所有 API 请求必须校验 school_id，禁止跨校拉取 POI。
异常逻辑：若用户不在任何围栏内，进入“游客模式”，仅展示最近的开放学校列表。
4.2 模块 B：高精度地图与 POI 众包
4.2.1 页面交互：长按地图点位触发“打点”；点击 POI 弹出抽屉式详情页，展示实时状态（拥挤度、是否开放）。
4.2.2 字段定义：poi_type(PGC/UGC), status_level(1-4级拥挤), expire_time(TTL)。
4.2.3 异常处理：恶意打点过滤（同一用户 1 小时内仅限新增 3 个 POI）。
4.3 模块 C：校内路线规划与实时导航（核心强化项）
4.3.1 精细化路径逻辑：
小路覆盖：支持高德公共地图未收录的“林间小道”、“建筑内部穿行路线”。
动态权重：支持设置“施工”、“阶梯（避开自行车）”、“坡度”等影响因素。
4.3.2 导航实时反馈：
偏航纠正：用户偏离预设路径 20 米时，自动重新发起 Recalculate。
位置平滑：针对校园高楼遮挡导致的 GPS 漂移，采用 Kalman 滤波进行位置平滑。
4.3.3 异常逻辑：弱网环境下，优先使用本地缓存的路径骨架。
五、 非功能性需求 (Non-Functional Requirements)
5.1 数据埋点需求 (Data Tracking)
导航行为埋点：nav_start (起点、终点), nav_repath (偏航次数), nav_finish (耗时)。
POI 互动埋点：poi_click, status_report (状态类型)。
5.2 性能与稳定性
首屏渲染：地图瓦片及 POI 标记加载需在 1.5s 内完成。
并发量：支持单校区同时 500 人在线导航。
5.3 安全与合规
权限控制：
L0-L1：只读地图。
L2 (认证学生)：有权发布众包信息、参与导航权重上报。
隐私保护：禁止存储并回传用户精确的实时位置点序列，仅允许上报脱敏后的“路径通行耗时”。
敏感词风控：众包 POI 的名称和描述必须经过内置敏感词库检索。
5.4 兼容性
移动端浏览器兼容：Chrome for Android, Safari iOS 最新 2 个版本。
六、 运营与上线准备 (GTM & Operations)
6.1 配置需求
管理员工作台：
围栏编辑器：可视化拖拽顶点定义校区范围。
POI 审核池：批量处理举报信息。
功能开关：支持按学校开启/关闭“实时导航”功能（针对未采点校区）。
6.2 灰度/发布策略
Alpha 阶段：开发者所在校区全量测试。
Beta 阶段：联系导员，邀请 10% 的学生进行“寻宝式”打点测试。
全量阶段：根据各校管理员维护进度，逐校解锁。
