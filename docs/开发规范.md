# Campus Survival Guide — 开发规范

本文档与本仓库 **`package.json`、`prisma/schema.prisma`、源代码** 一致；若与其它文档冲突，以**代码与 Prisma** 为准。

---

## 1. 文档目的与适用范围

| 目的 | 说明 |
|------|------|
| 统一范式 | Next.js App Router + TypeScript 全栈在**同一套规则**下开发 |
| 提升复用 | 业务逻辑抽取到 `lib/`，交互抽取到 `hooks/`，界面按域放入 `components/<domain>/` |
| 减少冗余 | **先检索再写**：禁止平行实现同一能力（两套 Modal、三套请求格式等） |
| 目录清晰 | **按职责放文件**，不搞「巨无霸 page」或散落在各处的魔法字符串 |
| 规则可执行 | PR / 自检清单可逐项勾选 |
| 控制复杂度 | 通过边界、拆分、单一验证路径降低「屎山」概率 |

---

## 2. 技术栈与运行时约定

- **框架**：Next.js **14（App Router）**、React 18  
- **语言**：TypeScript（**避免无必要的 `any`**；SDK 边界可局部收窄类型）  
- **样式**：Tailwind CSS；类名合并使用 **`cn()`**（`clsx` + `tailwind-merge`）  
- **数据**：Prisma + **MySQL**  
- **全局状态**：Zustand — **仅限跨页面/跨树的全局 concern**  
- **校验**：**Zod**（新入口建议统一 Schema + 推导类型）  
- **地图**：高德 AMap JS API 2.0，经 **`@amap/amap-jsapi-loader`** 加载  

---

## 3. 推荐目录结构与职责边界

以下为**约定**（与当前仓库基本一致）；新增代码应落在对应层级。

```
app/
  (routes)/                 # App Router：页面、layout、加载/错误边界
  api/**/route.ts           # Route Handler（HTTP、cron、需纯 HTTP 的集成）
components/
  ui/                       # 通用 UI（按钮、Modal、Drawer 壳等）
  <domain>/                 # 领域组件（market、poi-drawer、admin 等）
hooks/                      # 可复用的客户端副作用、订阅、小型状态机
store/                      # Zustand 等全局状态
lib/
  actions/                  # Server Actions："use server"、写库与变更
    <feature>/             # 大域可分子目录（如 comment/、category/）
  auth/                     # 鉴权、Cookie、JWT、中间件可读逻辑
  core/                     # prisma 封装等基础设施
  <domain>/                 # 纯函数、类型、常量（无 "use server"）
prisma/
  schema.prisma
```

**禁止：**

- 在 **Client Component** 内直接 **Prisma / 手写 SQL**。  
- 把整页业务垒在单个 **`page.tsx` 顶部**超过合理规模（参见第 11 节）。  
- 在不知道是否已有封装的情况下新建「第二套」同样的工具或 UI 壳。

## 4. Next.js 全栈范式（核心）

### 4.1 默认渲染模型

| 层级 | 建议 |
|------|------|
| **Server Components** | 首屏数据、无浏览器 API 的内容优先 RSC |
| **Client Components** | 仅在有 `useState` / 事件 / 浏览器 API / 地图等必要时加 **`"use client"`** |
| **边界** | 尽量把 `"use client"` 沉到**叶子组件**，避免整棵 layout 客户端化 |

### 4.2 数据变更与 HTTP

| 场景 | 首选 | 说明 |
|------|------|------|
| 表单 / 按钮触发的业务写入与页面数据加载 | **Server Action**（`lib/actions/**`、`lib/market/*`、`lib/school/actions.ts`） | **默认且唯一业务路径** |
| Cron、运维 seed、登出 Cookie 兜底 | **`app/api/**/route.ts`**（白名单见 `docs/API.md`） | 当前仅 4 个端点 |
| **禁止** | 同一业务规则同时在 Action 和 Route **各写一遍** | 新增 Route 须 PR 说明 HTTP 必要性 |

**HTTP 白名单（2026-05-19）：** `/api/cron/market-deadlock`、`/api/auth/logout`、`/api/auth/seed`、`/api/auth/seed-test-accounts`。

若某功能同时使用 **`fetch('/api/...')` 与 Server Action**（当前仅登出 fallback），须在相关文件顶部用**简短注释**说明原因。

### 4.3 路由与权限页

- 管理后台路径：由 **`middleware.ts`** 与业务内 **`AuthGuard`** 等与角色语义一致。  
- **中间件鉴权**：管理端路由使用 Cookie 中的 **JWT** 解析角色（无服务端 HTTP 自省）；与用户态、Server Action 侧的 **`getAuthCookie()` / `getMe()`** 等保持**角色语义一致**，避免「中间件能进、Action 报无权限」的漂移。

## 5. 多租户、鉴权与角色（必须遵守）

1. **`schoolId`**  
   - 凡表含 `schoolId` 的查询，默认 **`where: { schoolId: currentSchoolId }`**。  
   - **超级管理员跨校**必须**显式、注释、范围最小**。

2. **写入**  
   - **绝不信任**客户端传来的 `schoolId`。从 **`getAuthCookie()` / 会话**注入。  
   - 已知无 `schoolId` 的实体（如部分反馈类）在代码与产品文档中**注明**。

3. **角色**  
   - 数据库存**数字** `role`；项目内使用**唯一映射**（如 `roleToLabel`、`assertRole`）。  
   - **禁止**：`roleMap[x] || "STUDENT"` 等把 **0 / 未知**静默当成学生。未知角色须按产品规则：**GUEST / 抛错 / 401/403**，**禁止静默提权**。

4. **Server Action 中的 `redirect()`**  
   - 在 `try/catch` 中须 **rethrow** Next 的 redirect 错误，例如：  
     `if (error instanceof Error && error.message.includes("NEXT_REDIRECT")) throw error;`

## 6. 数据层与 API 契约

### 6.1 Prisma

- 复杂 `select` 优先 **`Prisma.XxxGetPayload<...>`** 或显式 interface。  
- 关键状态流转、防重复提交：在合适场景使用**事务**与**唯一约束**。

### 6.2 API 响应形状（功能内统一）

同一功能域内保持一种约定，推荐：

```ts
{ success: true, data: T, message?: string, pagination?: ... }
// 失败时 success: false + error / message（与本域已有写法一致）
```

**禁止**在同一业务线混用 `ok` / `success` 等无共识字段，除非经**全项目**决策后批量迁移。

### 6.3 分页

- 使用 `take` / `skip`；遵守**最大 page size**（如 100）。  
- `total` / `pageCount` 与列表长度**一致、可解释**。

## 7. 组件、Hooks 与复用

### 7.1 组件分层

| 类型 | 位置 | 职责 |
|------|------|------|
| 展示型 | `components/<domain>/` | props 驱动、少副作用 |
| 通用壳 | `components/ui/` | Modal、Drawer、PageState 等 |
| 页面编排 | `app/**/page.tsx` | 组合子组件，避免上千行业务逻辑 |

### 7.2 Hooks

- **地图、轮询、订阅、复杂 useEffect 链** → `hooks/use-*.ts`。  
- **禁止**在多个页面复制同一套 `useEffect` 依赖逻辑。

### 7.3 状态

- **Zustand**：auth、导航、过滤器等**真·全局**。  
- 单棵树内状态：**useState / useReducer / Context**，不滥用全局。

### 7.4 内容与审核

- UGC：复用 **`validateContent`** 等现有 `lib` 工具，**禁止**各处复制敏感词与长度规则。

---

## 8. UI / UX（Tailwind）

1. 类名：始终 **`cn()`** 拼接。  
2. **z-index**：Navbar **40**；Modal 遮罩 **100**、内容 **110**；Tooltip **120**；确有需求再 **z-[200]** 等与现有页面一致。  
3. **Modal / Drawer**：`max-h-[min(85vh,calc(100vh-40px))]`；正文 **`overflow-y-auto`**；头部/底部固定。  
4. **搜索框**：防抖 **`useDebounce`（建议 300ms）**。  
5. **列表/详情页**：至少具备 **loading / empty / error / 无权限** 之一组状态，或文件中**明确写明**为何不适用。

---

## 9. 地图专项（高德 AMap 2.0）

1. 仅通过 **`@amap/amap-jsapi-loader`** 加载；**不要**默认 `import AMap from ...`。  
2. Map / Marker / Polyline **实例放进 `useRef`**，避免每帧 new 导致 WebGL/上下文问题。  
3. 坐标系：**GCJ-02**。  
4. 与视觉一致的 Marker **`offset`** 等沿用项目既定值（如 POI Map 中所用）。  
5. **`any` / `unknown`** 限制在 SDK 边界，业务层用窄类型或小接口承接。

## 10. TypeScript、环境与工程质量

### 10.1 类型与错误

- 禁止把 `any` 传进核心业务；对外部 JSON 先做校验（Zod）再使用。  
- 异步按钮：**pending 时禁用**，防双提交。

### 10.2 环境变量

- **`process.env` 缺失时行为须定义**：开发可降级+log；生产 Fail fast 或明确日志。  
- **禁止**编造 cookie 名、表名；从 **`lib/auth/server-actions.ts`、`prisma/schema.prisma`、真实 route** 溯源。

### 10.3 依赖

- 新增 npm 包前：先查 **`package.json`** 是否已有同类能力。

### 10.4 本地检查

- 合并前建议：`npm run lint`、`npx tsc --noEmit`（与 CI 策略对齐）。

## 11. 命名与导出约定（建议）

| 类别 | 约定 |
|------|------|
| 页面文件 | Next 约定：`page.tsx`、`layout.tsx`、`loading.tsx`、`error.tsx` |
| React 组件 | PascalCase：`UserProfileModal` |
| hooks | `useXxx`，文件 `use-xxx.ts` |
| Server Actions | 动词开头、`camelCase`，与现有 `lib/actions` 保持一致 |
| 类型 | 共用类型放 `lib/<domain>/types.ts` 或 Prisma Payload |
| Barrel `index.ts` | 按需导出；**注意循环依赖**，兼容层命名避免与文件夹同名打架 |

## 12. Git 与评审清单（自检）

- [ ] **多租户**：读/写是否按 `schoolId` 收口；超管跨校是否显式且收窄

- [ ] **校验**：Action 与 Route（若并存）是否共享一套校验辅助  
- [ ] **重复**：检索过是否仍存在平行实现  
- [ ] **文件体积**：是否需要拆 `lib`/hooks/components  
- [ ] **`any`**：是否仅出现在 SDK/边界
