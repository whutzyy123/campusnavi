# 潜在风险修复计划

本文档对应《代码审查报告》**## 三、潜在风险（当前未必已触发故障）** 中的四项。

**性质说明**：本节问题未必已在线上表现为故障；计划侧重 **预防性治理**（可观测性、成本控制、交付纪律）。可与 [P1 修复计划](./P1修复计划.md)、[P2 修复计划](./P2修复计划.md) **并行**，但以下内容与 P1/P2 **交叉时**须在 PR 中单点说明优先级。

---

## 总览与推荐顺序

| 顺序 | 风险项 | 触发条件 | 建议节奏 |
|:---:|:---|:---|:---|
| 1 | 重构分支体量过大 | 合并/上线窗口 | **立即**：合并策略与评审颗粒度 |
| 2 | `RateLimit` 表膨胀 | 长期在线 + 高频限流键 | **短周期**：清理或轮转策略 |
| 3 | `force-dynamic` 面过大 | 流量上升、账单敏感 | **中周期**：逐个路由复盘是否必须动态 |
| 4 | 缺乏自动化测试 | 每次大范围改动 | **持续**：先有「最小护栏」再扩展 |

说明：表中 1 不依赖编码即可部分缓解；3、4 可长期迭代。

---

## R-1：缺少自动化测试护栏

### 风险回顾

未发现单元/集成/e2e 测试文件；大规模重构或合并时回归高度依赖手工。

### 目标

建立 **低风险、可复制** 的测试底座：**先保命（鉴权与契约）**，再扩业务覆盖面。

### 实施步骤（分阶段）

**阶段 A：基建（1 个 PR）**

- 选型与版本锁定：`vitest` + `@testing-library/react`（或与 Next 官方推荐一致的一套），写入 `package.json` 脚本，例如：
  - `test` / `test:ci`
- 约定测试目录：`__tests__/` 或与源码并列 `*.test.ts`（选定一种写入 `docs/开发规范.md`）。
- CI（若有）：在流水线中运行 `npm run test:ci`，失败阻断合并。

**阶段 B：纯函数与高价值模块（可多 PR）**

- 首批覆盖不涉及 DB 的逻辑：`lib/auth/role.ts`（`dbRoleToAppRole`）、`lib/core/utils.ts`（分页裁剪）、content 校验辅助等。
- 目标：**稳定、无副作用、易断言**。

**阶段 C：需要 DB 的集成测试（按需）**

- 使用 `.env.test` + 独立数据库或 Docker MySQL，`prisma migrate` / `db push` 在 CI 中单次执行。
- 覆盖：`getAuthCookie`/`requireSchoolAdminJson` 类行为（或通过 Route Handler supertest）；**租户隔离断言**（`schoolId` 不可串校）。

**阶段 D：E2E（可选，流量路径稳定后）**

- Playwright 等：登录 → 主页 → 集市关键路径冒烟；低频运行（如 nightly）降低维护成本。

### 验收标准

- 主干存在可一键执行的 `npm run test`（或等价），且 CI 中执行。
- 至少 **≥5 条**有意义的单测或可重复的集成断言（阈值可按团队调高）。
- 「关键改动清单」（鉴权、限流、cron、`prisma/schema`）要求在 PR 中注明是否增补/更新了测试。

### 与 P1/P2 关系

P1「中间件/W鉴权收口」完成后，优先考虑为 **会话解析共用函数** 补测试，避免 Middleware 与 `getAuthCookie` 漂移无自动化发现。

## R-2：限流表（`rate_limits`）增长的运维风险

### 风险回顾

`consumeRateLimit` 基于 Prisma `RateLimit` 表固定窗口计数；未见明确过期行清理，`key` 随业务维度持续增长。

### 目标

在长期运行下 **限制表行数与索引工作量**，不改变限流语义或引入明显额外延迟。

### 方案选项（选一为主，可做组合）

**方案 A：定时批量删除陈旧窗口**

- 依据：`window_start` 早于「当前时间 − 若干倍最大窗口」（例如 `(max(windowMs))*2`）。
- 实现：`cron` 路由或服务端定时脚本 `DELETE WHERE window_start < ? LIMIT ?`（分块删除防锁表）。
- 注意：须在代码中枚举或配置实际使用的最大 `windowMs`，避免删过头影响当前窗口）。

**方案 B：迁移到 Redis**

- `ioredis` 已在 `package.json` 依赖中可考虑复用：`INCR` + `PEXPIRE` TTL，天然不占 MySQL 行数。
- 成本：运维与一致性（多实例时钟、故障降级）。适合流量与实例数较高的阶段。

**方案 C：DB 归档 + 保留最近 N 天**

- 保留策略与审计需求对齐。

### 实施步骤

1. 统计生产/预发：**当前行数、`key` 前缀分布、`window_start` 最大值与最小值**。
2. 选择方案并与 P1 「cron 密钥」/`CRON_SECRET` 惯例对齐新增清理任务路由（若为 HTTP cron）。
3. 加 **监控告警**：表行数、清理任务最近一次成功时间。
4. 文档：`docs/` 或运维手册中写明「为何限流在 DB、清理周期、回滚」。

### 验收标准

- 在压测或模拟长期运行环境中，确认表行数有 **上界或可预期回落**。
- 限流失败后业务表现与文档一致（429 文案等）。

## R-3：`force-dynamic` 使用范围过大

### 风险回顾

多个 Route Handler 显式 `dynamic = 'force-dynamic'`，弱化边缘/全页缓存潜力，吞吐与计费对流量波动更敏感。

### 目标

对每个标记 **有理由可查**：必须动态则说明原因；能不动的改为默认或按需 `revalidate`。

### 实施步骤

1. **清单化**：全仓 `grep "force-dynamic"` 列出文件，按「是否真的依赖 `cookies`/`headers`/`searchParams`/每次请求都不同」分组。
2. **分类处置**：
   - **依赖请求头 Cookie 或 `getAuthCookie` 链路**：多半必须动态，在文件顶端 **一行注释原因**。
   - **仅因历史习惯**：尝试移除常量，依托 Next 默认；若出现问题再恢复并注释。
   - **读多写少**：考虑 `fetch`/`unstable_cache` + `tags`（视版本与兼容性）或上移为 Server Component 静态段。
3. **分 PR 改动**：每次只动一类路由，避免大范围行为漂移。
4. **观测**：staging 上对关键接口打点 TTFB、p99；与移除前对比。

### 验收标准

- 每个剩余 `force-dynamic` 处有简短注释链接到需求（Issue/PR）。
- 对「误判可静态」的路由：**已验证**在无 Cookie 或可缓存语义下仍能正确放行/拒绝。

### 注意

与用户登录态绑定的 Handler **不应强行去动态化**，否则可能引入安全风险；本项优先考虑 **可读性、成本文档化** 与安全边界。

## R-4：重构阶段分支/工作树差异过大

### 风险回顾

大量修改、删除、未跟踪文件跨 `app`、`components`、`lib`、`docs`，合并冲突与隐性回归概率高。

### 目标

将「大重构」拆解为 **可评审、可回滚、可按域合并** 的线性或小批分支；降低单次合并 Cognitive load。

### 实施步骤（流程为主，辅以工具）

**合并与分支策略**

- 禁止超大面积单 PR：按域拆分，例如：`lib/market`、`app/api/admin`、`docs`。
- **竖切优先**：单一用户故事端到端可走通，而非横向改完所有文件名再收尾。
- 每个 PR：**明确回滚开关**（特性开关）或限定影响面。

**Git 与工作区纪律**

- 定期 `git status` 清点 `??`，决定纳入 `.gitignore` 或入库（如误生成的 `.next` 缓存）。
- **删除文件**在同一 PR 中说明替代路径（避免「删了旧的、新的还未合入」断层）。

**评审**

- 大 PR：要求自检清单（鉴权、`schoolId`、双倍提交）（与 `.cursor/rules` Pre-merge 一致）。
- **变更说明**：自动生成 `git diff --stat` 附上，便于指派第二评审人只看高风险路径。

### 验收标准

- 主干合并节奏稳定：**无单次 PR 触碰 >N 文件**的团队约定（可按项目定 N，如 80）或有拆 PR 的书面例外说明。
- 冲突热点目录（如 `middleware`、`prisma/schema`）改动单独成 PR。

### 与 P1/P2 关系

P1 「入口收口、契约统一」本质是减少长期分支分叉；与本节流程结合可显著降低隐性回归。

## PR / 运维自检（潜在风险视角）

- [ ] 高频写表（限流日志等）是否有增长边界或告警。
- [ ] 路由是否不必要的 `force-dynamic`。
- [ ] 破坏性合并是否拆分；高风险路径是否双人审阅。
- [ ] 本轮是否增补/调整了自动化测试或可重复手工脚本。
